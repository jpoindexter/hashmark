# Hashmark GitHub Action — Technical Specification

## Overview

The Hashmark GitHub Action is the core delivery mechanism for Pro and Team tiers. It runs in the user's own CI environment, scans their codebase with hashmark's 27 scanners, generates all 7 AI context file formats, and auto-commits changes to the repo.

**Key design principle**: Zero friction. No PRs to review. No manual steps. Context files are always up-to-date.

## Action Definition

### `action.yml`

```yaml
name: 'Hashmark — AI Context Sync'
description: 'Auto-generate and sync AI context files for every coding tool'
branding:
  icon: 'hash'
  color: 'green'

inputs:
  formats:
    description: 'Comma-separated list of formats to generate (default: all)'
    required: false
    default: 'agents,claude,cursorrules,cursor-mdc,copilot,windsurf,gemini'
  mode:
    description: 'Delivery mode: "commit" (auto-commit) or "pr" (create PR)'
    required: false
    default: 'commit'
  compact:
    description: 'Use compact output mode (smaller files)'
    required: false
    default: 'false'
  custom-rules:
    description: 'Path to custom rules file (hashmark.config.json)'
    required: false
    default: ''
  exclude:
    description: 'Glob patterns to exclude from scanning'
    required: false
    default: ''

outputs:
  scan-summary:
    description: 'Summary of what was scanned'
  files-changed:
    description: 'Number of files that were updated'
  formats-generated:
    description: 'List of formats that were generated'

runs:
  using: 'composite'
  steps:
    - name: Setup Node.js
      uses: actions/setup-node@v4
      with:
        node-version: '22'

    - name: Run Hashmark scan
      shell: bash
      run: |
        npx hashmark-cli . --json --force
        # Parse JSON output
        # Generate all requested formats
        # Determine which files changed

    - name: Commit or PR
      shell: bash
      run: |
        # Based on mode input:
        # commit: git add + git commit + git push
        # pr: create branch + commit + open PR
```

## Generated Workflow File

When a user clicks "Install Action" in the Hashmark web UI, we create this file in their repo via GitHub Contents API:

### `.github/workflows/hashmark.yml`

```yaml
name: Hashmark — Sync AI Context

on:
  push:
    branches: [main, master]
    # Don't trigger on context file changes (prevent infinite loop)
    paths-ignore:
      - 'AGENTS.md'
      - 'CLAUDE.md'
      - '.cursorrules'
      - '.cursor/rules/**'
      - '.github/copilot-instructions.md'
      - '.windsurfrules'
      - 'gemini.md'

  # Allow manual trigger
  workflow_dispatch:

permissions:
  contents: write

jobs:
  sync:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Full history for git scanner

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'

      - name: Run Hashmark
        run: |
          npx hashmark-cli . --json --force > /tmp/scan-results.json

      - name: Generate all formats
        run: |
          # This step uses the scan results to generate all 7 formats
          # Each format has its own generator that transforms scan data
          # into the tool-specific format

          node -e "
            const fs = require('fs');
            const results = JSON.parse(fs.readFileSync('/tmp/scan-results.json', 'utf8'));

            // AGENTS.md — already generated by hashmark
            // Copy from scan output

            // CLAUDE.md — transform to Anthropic's format
            // .cursorrules — transform to Cursor legacy format
            // .cursor/rules/project.mdc — transform to Cursor MDC format
            // .github/copilot-instructions.md — transform to Copilot format
            // .windsurfrules — transform to Windsurf format
            // gemini.md — transform to Gemini CLI format
          "

      - name: Check for changes
        id: changes
        run: |
          git diff --quiet || echo "changed=true" >> $GITHUB_OUTPUT

      - name: Commit changes
        if: steps.changes.outputs.changed == 'true'
        run: |
          git config user.name "hashmark[bot]"
          git config user.email "bot@hashmark.md"
          git add AGENTS.md CLAUDE.md .cursorrules .cursor/rules/ .github/copilot-instructions.md .windsurfrules gemini.md
          git commit -m "chore: sync AI context files [hashmark]

          Updated by Hashmark (hashmark.md)
          Formats: AGENTS.md, CLAUDE.md, .cursorrules, .cursor/rules/project.mdc, copilot-instructions.md, .windsurfrules, gemini.md"
          git push
```

## Format Generators

Each format requires a specific transformation of the scan results:

### 1. AGENTS.md (Universal)

Already generated by hashmark CLI. This is the source format.

**Structure**:
```markdown
# AGENTS.md
> Auto-generated by Hashmark

## TL;DR
- Stack, component count, key imports, high-impact files

## Getting Started
- Auto-detected setup commands

## Critical Rules
- WRONG/RIGHT code examples based on detected patterns

## Components
- Full inventory grouped by category

## API Routes
- Routes with methods, auth status, schemas

## Database Models
- Fields and relations

## Environment Variables
- Required vs optional

## Code Patterns
- Detected patterns with examples

## Design Tokens
- Color tokens with usage guidance
```

### 2. CLAUDE.md (Claude Code)

**Differences from AGENTS.md**:
- Includes `## Commands` section (npm scripts) prominently at top
- Uses Anthropic's recommended structure (project overview → commands → architecture → patterns)
- Includes file paths more prominently (Claude Code reads these for navigation)
- Adds `## For Claude Code` section with specific instructions

**Structure**:
```markdown
# CLAUDE.md

## Project Overview
[Framework] + [Language] + [Key dependencies]

## Commands
npm scripts with descriptions

## Architecture
Directory structure with descriptions

## Key Patterns
Design patterns, naming conventions, import conventions

## Critical Rules
WRONG/RIGHT examples

## Components
Component inventory with import paths

## API Routes
Routes with schemas

## Database
Models with fields
```

### 3. .cursorrules (Cursor Legacy)

**Differences**:
- Flat text format (no markdown headers)
- Rule-oriented (imperative statements)
- Shorter, more concise
- Focused on what to DO and NOT DO

**Structure**:
```
You are working on a [framework] project with [language].

Always:
- Use existing components from [path]
- Follow [pattern] for [thing]
- Use design tokens, not hardcoded colors

Never:
- Create new components without checking existing ones
- Use raw HTML when a component exists
- Hardcode colors or spacing values

Components available: [list]
API routes: [list]
Database models: [list]
```

### 4. .cursor/rules/project.mdc (Cursor MDC)

**Differences**:
- MDC format with YAML frontmatter
- Can include `globs` for file-specific rules
- Richer metadata (description, tags)

**Structure**:
```markdown
---
description: Project context for AI assistance
globs: ["**/*.tsx", "**/*.ts"]
---

# Project Rules

## Stack
[Framework details]

## Components
[Component inventory with paths]

## Patterns
[Code patterns with examples]
```

### 5. .github/copilot-instructions.md (GitHub Copilot)

**Differences**:
- Specifically for GitHub Copilot Chat and inline suggestions
- Should reference GitHub-specific concepts (PRs, issues)
- Shorter format, Copilot has smaller context window

**Structure**:
```markdown
# Copilot Instructions

## Project
[Brief description]

## Tech Stack
[Key technologies]

## Conventions
- [Convention 1]
- [Convention 2]

## Components
[Key components to reuse]

## Don't
- [Anti-pattern 1]
- [Anti-pattern 2]
```

### 6. .windsurfrules (Windsurf)

**Differences**:
- Simple rules format similar to .cursorrules
- Codeium/Windsurf specific conventions

**Structure**:
```
Project: [name]
Framework: [framework]
Language: [language]

Rules:
1. [Rule 1]
2. [Rule 2]
...

Components: [list]
Patterns: [list]
```

### 7. gemini.md (Gemini CLI)

**Differences**:
- Markdown format, similar to CLAUDE.md
- Optimized for Google's Gemini models
- Can be more verbose (Gemini has large context windows)

**Structure**:
```markdown
# Project Context

## Overview
[Detailed project description]

## Architecture
[Full architecture with file paths]

## Components & APIs
[Comprehensive inventory]

## Rules
[Coding rules with examples]
```

## Infinite Loop Prevention

Critical: The GitHub Action must NOT trigger itself. If the action commits updated context files, that push would trigger the action again, creating an infinite loop.

**Solution**: `paths-ignore` in the workflow trigger:
```yaml
on:
  push:
    paths-ignore:
      - 'AGENTS.md'
      - 'CLAUDE.md'
      - '.cursorrules'
      - '.cursor/rules/**'
      - '.github/copilot-instructions.md'
      - '.windsurfrules'
      - 'gemini.md'
```

**Backup solution**: Check commit author:
```bash
if [ "$(git log -1 --format='%ae')" = "bot@hashmark.md" ]; then
  echo "Skipping — last commit was from Hashmark"
  exit 0
fi
```

## Configuration File

Users can optionally create `hashmark.config.json` in their repo root:

```json
{
  "formats": ["agents", "claude", "cursorrules", "cursor-mdc", "copilot", "windsurf", "gemini"],
  "mode": "commit",
  "compact": false,
  "exclude": [
    "**/test/**",
    "**/fixtures/**",
    "**/stories/**"
  ],
  "customRules": [
    {
      "name": "Use design tokens",
      "rule": "Always use semantic color tokens (bg-primary, text-foreground) instead of hardcoded colors (bg-blue-500, text-white)"
    },
    {
      "name": "Component reuse",
      "rule": "Check existing components before creating new ones. The project has 70+ reusable components."
    }
  ]
}
```

## Metrics to Collect

The GitHub Action should report back to Hashmark (optional, with user consent):

- Scan duration
- Number of files scanned
- Number of formats generated
- Number of files changed
- Commit SHA
- Error messages (if scan fails)

This enables the Hashmark dashboard to show live scan status without requiring a webhook.

## Rate Limiting / Abuse Prevention

- Free tier: No GitHub Action (manual scan only via web UI)
- Pro tier: Unlimited Action runs (runs in user's own CI minutes, no cost to us)
- API calls to Hashmark backend: Rate limited to 100/hour per user
- Scan results stored for 90 days (Pro), 30 days (Free)
